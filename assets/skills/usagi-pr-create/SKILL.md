# PR を新規作成する

PR を新しく作るときの手順。プログラミング言語やフレームワークに依存しない、どのプロジェクトでも共通のルールをまとめる。

## 1. 本文を書く言語を決める

- タイトル・本文は **依頼してきた人（チャットの相手）が使っている自然言語と同じ言語**で書く。会話が日本語なら日本語、英語なら英語、それ以外でも相手に合わせる。
- 言語の判定は、直近のやり取りで相手が実際に使っている言語で行う。プロジェクトの既定言語やこのスキルの記述言語ではなく、**相手に合わせる**。
- 以降の見出し名（「何を実行したか」など）も、決めた言語へ訳して使う。

## 2. 依存する前段の PR があるか確認する

このPRが別の未マージ PR のブランチ上に積まれている、またはその変更に依存している場合は、本文の **いちばん先頭**に依存とマージ順を明記する。

```
> ⚠️ このPRは #<番号> に依存しています。**先に #<番号> をマージしてから**このPRをマージしてください。
```

- 依存先が複数ある場合はすべて挙げ、マージすべき順序を書く。
- 独立した PR（前段なし）ならこの節は付けない。

## 3. 本文を組み立てる

本文は次の 4 つの見出しで構成する（相手の言語に訳す）。各見出しは省略しない。

### 何を実行したか
このPRで行った作業を 1〜数行で要約する。読み手が最初の一読で「何のPRか」を掴めるようにする。

### なぜ
その変更が必要になった背景・目的・解決する課題を書く。関連する issue / チケットがあればここでリンクする。

### 変更点
主要な変更を箇条書きで列挙する。ファイル単位の羅列ではなく、**振る舞い・仕様の観点**でまとめる。破壊的変更・移行が必要な点があれば明示する。

### 動作確認
- テスト・Lint・ビルドなど、そのプロジェクトの品質チェックを実行し、**結果（成功した旨や出力）**を載せる。
- UI に変更がある場合は、変更後の画面の**スクリーンショットまたはイメージ図を添付**する。Before/After があると望ましい。画像を自分で添付できない場合は、どの画面のどの図を貼るべきかを本文に明記し、相手に添付を促す。

## 4. PR を作成する

ブランチを push してから PR を作る。タイトルはプロジェクトの規約（Conventional Commits 等）に合わせる。

```bash
git push -u origin <branch>
gh pr create --title "<タイトル>" --body "<上の構成で組み立てた本文>"
```

本文を複数行で渡すときは `--body-file` にファイルを渡すか、`$'...\n...'` で改行を入れる。

## 完了チェック

- [ ] 本文は相手の言語で書かれている
- [ ] 依存する前段 PR があれば、先頭にマージ順を明記した
- [ ] 「何を実行したか / なぜ / 変更点 / 動作確認」がすべて埋まっている
- [ ] 動作確認に実行結果を、UI 変更があれば画像を載せた
