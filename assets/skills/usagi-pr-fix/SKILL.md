# PR を修正する

レビュー対応・最新化・コンフリクト解消をまとめて行い、PR を仕上げる手順。プログラミング言語に依存しない共通ルール。最後に必ず `usagi-pr-update` スキルを実行する。

## 前提: gh が認証で使えないとき

このスキルの手順は `gh` を使う。まず `gh auth status` を確認し、通っていればそのまま `gh` を使う。未認証・認証エラーで使えない場合は、既に `GH_TOKEN` または `GITHUB_TOKEN` が環境変数として渡されていないか確認する。どちらも無ければ、ユーザーに `gh auth login` を促して中断する（トークンを勝手に探さない）。

環境変数でトークンが渡されている場合は、そのまま `gh` を実行する。

   ```bash
   # GH_TOKEN を安全な方法で環境変数にセット済みの状態で実行する
   gh pr view <番号> --comments
   gh api repos/{owner}/{repo}/pulls/<番号>/comments
   ```

- トークンは PR 本文・コメント・コミット・ログ・コマンド履歴に残さない。値をそのまま貼り付けたり表示したりしない。

## 1. レビューコメントに対応する

PR に付いているレビュー（コメント・変更要求）を取得し、内容を直す。

```bash
gh pr view <番号> --comments
gh api repos/{owner}/{repo}/pulls/<番号>/comments   # 行コメントも含めて確認
```

- 各コメントが求めている変更をコードに反映する。
- 対応しないと判断したものは理由を整理しておく（返信は手順 4 の `usagi-pr-update` で行う）。

## 2. 最新化する（ベースブランチに追従）

ベースブランチ（通常 `main`）の最新を取り込み、PR ブランチを最新の状態にする。

```bash
git fetch origin
git rebase origin/main      # 履歴を直線に保つ場合
# もしくは方針に応じて: git merge origin/main
```

- プロジェクトの方針（rebase / merge）に合わせる。不明なら既存の PR 運用に倣う。

## 3. コンフリクトを解消する

最新化でコンフリクトが出たら解消する。

```bash
git status                  # コンフリクトしたファイルを確認
# 各ファイルを編集して <<<<<<< ======= >>>>>>> を解消
git add <解消したファイル>
git rebase --continue       # rebase の場合（merge なら git commit）
```

- 両方の変更の意図を汲んで統合する。どちらか一方を機械的に捨てない。
- 解消後は、そのプロジェクトの品質チェック（テスト・Lint・ビルド）を通して壊れていないことを確認する。

## 4. 反映して PR を仕上げる

修正をコミットし、push する。rebase で履歴を書き換えた場合は force-with-lease を使う。

```bash
git push --force-with-lease   # rebase した場合
# rebase していなければ通常の git push
```

そのうえで **`usagi-pr-update` スキルを実行する**。これにより概要が最新化され、各レビューコメントへの返信まで完了する。

## 完了チェック

- [ ] レビューコメントが求める変更を反映した
- [ ] ベースブランチの最新を取り込んだ
- [ ] コンフリクトを解消し、品質チェックが通る
- [ ] 修正を push した
- [ ] 最後に `usagi-pr-update` スキルを実行した（概要の最新化・レビュー返信）
